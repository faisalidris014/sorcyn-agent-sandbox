### Known Limitations

- **Pre-existing chip/micro-element radii outside {12, 16, 24}**: ~80
  occurrences across the 39 screens (mostly 6, 8, 9, 10, 20 px on chip
  pills, summary-card icon containers, trend pills, and chart bars).
  These follow Material Design chip / pill conventions and pre-date the
  GSD planning bootstrap. The audit script reports them as informational
  tech debt; they do not fail the gate. Suggested follow-up: a Phase 2.x
  polish plan that either (a) adds inline justification comments to each
  occurrence per truth #5, or (b) defines an additional locked token
  (e.g. `chipRadius = 20`) and re-runs the audit.
- **Ad-hoc hex colors outside the locked palette allow-list** in
  ~12 screens (Tailwind blues on dispute-related avatars, ambers on
  stripe-onboard / verification, an indigo on payment-methods, etc.).
  Same disposition: pre-existing baseline state, surfaced for follow-up,
  not blocking.
- **`tap_scale.dart`** has no production callers yet — by design (existing
  button widgets embed their own gesture/scale logic and were not
  refactored to avoid regression risk on shipped buttons). Future ad-hoc
  tappable consumers should use `TapScale`.
- **`gradient_fab.dart`** is consumed in only one place (the buyer
  dashboard layout). Phase 3 may add seller-side dashboard FABs.
- **Splash screen** restyle is handled natively
  (`flutter_native_splash`), not in Flutter widget code. Per
  `BATCH_8_PLAN`, native splash assets are out of scope for this phase.
- **BATCH_1 plan file** is not present in the repository
  (`FRONTEND_RESTYLING_PLAN.md` references batches 1–8; the ingested
  files cover BATCH_2 through BATCH_8). The codebase audit (committed
  with the GSD planning bootstrap) confirmed BATCH_1 deliverables —
  theme + Inter font + auth screens + foundation shared widgets
  (`gradient_button`, `social_auth_button`, `app_input_field`,
  `app_logo`) — are all shipped at commit `804c8ec`. **This is the
  resolution of the STATE.md concern "BATCH_1 plan file is missing":
  documentation-only gap, no functional gap.**
- **Integration smoke test scope**: 5 `testWidgets` in
  `regression_smoke_test.dart`; 2 of them are pump-driven (LoginScreen-
  and HelpSupport-equivalent), 3 are constructor-only (Conversations,
  SubmitReview, and the Auth+ManualPost+SubmitOffer triple). Constructor-
  only tests are explicitly documented in-test as a deliberate compromise
  — those screens depend on Riverpod providers / socket timers that
  cannot be cleanly mocked at this layer. Full E2E coverage lives in
  Phase 4.

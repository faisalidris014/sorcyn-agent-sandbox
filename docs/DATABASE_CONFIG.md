# Database Configuration — Authoritative Guide

> Single source of truth for how Sorcyn's databases are wired locally and how dev,
> test, migration, and seed paths each connect. If anything here disagrees with
> another doc, **this file wins** — update the other doc.
>
> Born out of issue #133: the four standard seed accounts kept vanishing from the
> local dev DB because the test suite ran destructive `DELETE FROM users` against
> the **same** database used for development. That can no longer happen — see
> [Test isolation guarantee](#test-isolation-guarantee).

---

## TL;DR

```bash
cd backend
cp .env.example .env          # fill in DATABASE_URL + secrets (first time only)
npm run db:bootstrap          # creates dev + test DBs, migrates, seeds, verifies
npm run db:doctor             # fast health check before any sim session
```

- **Dev DB:** `reverse_marketplace` — driven by `DATABASE_URL` in `backend/.env`.
- **Test DB:** `reverse_marketplace_test` — **derived** from the dev URL, driven by
  `backend/.env.test` (gitignored, generated by `db:bootstrap`).
- Tests **cannot** touch the dev DB: a guard refuses to run unless `NODE_ENV=test`
  **and** the DB name ends in `_test`.

## Connection-path map

Which code path connects to which database, and what drives it:

| Path | Driving env var | Database | How the env loads |
|------|-----------------|----------|-------------------|
| **Dev app runtime** (`npm run dev`) | `DATABASE_URL` | `reverse_marketplace` | `src/config/env.ts` loads `backend/.env` |
| **Migrations** (`prisma migrate …`) | `DATABASE_URL` | `reverse_marketplace` | `prisma.config.ts` runs `dotenv.config()` → `datasource.url` |
| **Seed** (`npm run db:seed`) | `DATABASE_URL` | `reverse_marketplace` | `prisma/seed.ts` runs `dotenv.config()` |
| **Tests** (`npm test`) | `DATABASE_URL` (with `NODE_ENV=test`) | `reverse_marketplace_test` | `vitest.config.ts` + `tests/global-setup.ts` load `backend/.env.test` (`override:true`) |
| **db:doctor** | `DATABASE_URL` | `reverse_marketplace` (dev) | `scripts/db-doctor.ts` runs `dotenv.config()` (loads `.env`) |

Why the test path wins over the dev `.env`: `vitest.config.ts` loads `.env.test`
with `override:true` **before** `src/config/env.ts` runs. `env.ts` then loads the
dev `.env` with dotenv's default `override:false`, so it can't clobber the test
values already in `process.env`.

## Fresh clone → working login

```bash
git clone <repo> && cd ReverseMarketplace/backend
cp .env.example .env
# Edit .env: set DATABASE_URL to match your local Postgres (see comments in the file).
#   Homebrew/native macOS:  postgresql://YOUR_OS_USER@localhost:5432/reverse_marketplace
#   Docker Postgres:        postgresql://postgres:postgres@localhost:5432/reverse_marketplace
npm install
npm run db:bootstrap   # creates both DBs, migrates, applies search trigger, seeds, verifies
npm run dev            # backend up on :3000
```

`db:bootstrap` is idempotent — re-run it any time. It:
1. Reads `DATABASE_URL` from `.env` (the single source of truth).
2. Creates `reverse_marketplace` and `reverse_marketplace_test` if missing.
3. **If it just created the dev DB**, syncs its schema with `prisma db push`.
   An **existing** dev DB is left untouched (see warning below). Then applies the
   full-text-search trigger and seeds (idempotent).
4. Generates `.env.test` (db name → `_test`, same user/host, `NODE_ENV=test`,
   Redis logical DB `/1`).
5. Syncs the test DB schema with `prisma db push --accept-data-loss` (test data
   is disposable, so it always converges to `schema.prisma`).
6. Runs `db:doctor`.

> **Which mechanism is authoritative where:** local dev + CI provision with
> **`prisma db push`** (fast, no migration authoring needed) — `schema.prisma` is
> the source of truth for the dev/test schema. **Production** applies the committed
> migrations in `prisma/migrations/` with **`prisma migrate deploy`**.
>
> As of issue #153 these two paths converge. The migration history was
> **rebaselined**: the eight prior migrations — which had drifted from
> `schema.prisma` in both directions (surfaced fixing #133) — were squashed into a
> single `00000000000000_baseline` generated directly from `schema.prisma`. So
> `migrate deploy` on an empty DB now reproduces `schema.prisma` **exactly**. This
> is proven on every PR by the **`migration-drift`** CI job (apply migrations to a
> clean DB → `migrate diff` must report zero drift), which also gates the
> production deploy. New schema changes ship as additive migrations going forward
> (forward-compatible-only rule, #26).
>
> ⚠️ Bootstrap still pushes the dev schema **only when it creates the dev DB**, and
> leaves an existing dev DB untouched. A dev DB built by earlier `db push` runs can
> still diverge from the current `schema.prisma`; if `db:doctor` flags it, recreate
> it (`DROP DATABASE reverse_marketplace`, then re-run bootstrap) rather than
> forcing a destructive push.

password hash, status, and verified flag of an existing account (previously the

by re-seeding — issue #133).

## Shared Supabase dev DB (issue #214)

Both operators (Faisal & Mohamed) share **one hosted Supabase dev database** so
their app data no longer drifts apart. The split that makes this safe:

| Path | Source of the URL | Database |
|------|-------------------|----------|
| **Dev app runtime** (`sorcyn dev`) | `DATABASE_URL` in **shared Doppler `dev` config** | shared Supabase (pooled) |
| **Migrations** (`prisma migrate deploy`) | `DIRECT_DATABASE_URL` in **Doppler** | shared Supabase (direct) |
| **Tests** (`npm test`) | `DATABASE_URL` in **local `.env` / `.env.test`** | **local** `reverse_marketplace_test` |

**Why tests stay local:** the suite runs destructive `DELETE FROM users` and
reseeds on every run (see [Test isolation guarantee](#test-isolation-guarantee)).
Pointing that at the shared DB would wipe the other operator's data and race
between machines. So tests **always** use local Postgres.

### How the URLs resolve without colliding

- The Supabase URLs live in the **shared Doppler `dev` config**, never in `.env`.
  `sorcyn dev` wraps the app in `doppler run`, which injects `DATABASE_URL` into
  the process. `src/config/env.ts` then loads `.env` with dotenv's default
  `override:false`, so the local `.env` can't clobber the Doppler value — Supabase
  wins for dev runtime.
- **Migrations** need a direct (non-pooler) connection: Supabase's transaction
  pooler (`:6543`, pgbouncer) doesn't support the advisory locks / prepared
  statements Prisma migrations use. `prisma.config.ts` therefore prefers
  `DIRECT_DATABASE_URL` (`:5432`) when set, falling back to `DATABASE_URL` for
  local Postgres where the two are the same host.
- **`.env` stays pointed at LOCAL Postgres.** `db:bootstrap` derives the test DB
  by appending `_test` to `.env`'s `DATABASE_URL` and runs destructive pushes
  against it, so it now **refuses to run** if `.env` points at Supabase
  (`pooler.` / `supabase.co|com`). Keep your local URL in `.env`.

`src/config/database.ts` auto-enables SSL when the URL contains `supabase.com`,
`sslmode=require`, or `NODE_ENV=production`.

### One-time provisioning (whoever owns the Supabase project)

```bash
# 1. Set the shared URLs in Doppler (NOT .env) — both operators inherit them.
doppler secrets set DATABASE_URL='postgresql://postgres.[project]:[pw]@aws-[n]-[region].pooler.supabase.com:6543/postgres?sslmode=require'
doppler secrets set DIRECT_DATABASE_URL='postgresql://postgres:[pw]@db.[project].supabase.co:5432/postgres?sslmode=require'

# 2. Apply schema + seed against the shared DB (run once).
cd backend
doppler run -- npx prisma migrate deploy   # uses DIRECT_DATABASE_URL
doppler run -- npm run db:seed             # 4 seed accounts + 38 categories

# 3. Day to day, both operators just:
doppler run -- npm run dev                 # or: sorcyn dev
```

## Test isolation guarantee

**Tests can never touch dev data.** Enforced by `backend/tests/db-guard.ts`
(`assertTestDatabase`), called before any destructive query in
`tests/global-setup.ts` and `tests/helpers.ts`. It throws unless **both**:

1. `process.env.NODE_ENV === 'test'`, and
2. the target database name ends in `_test`.

So even a misconfigured `.env.test` pointed at `reverse_marketplace` aborts the
run instead of wiping seed accounts. To prove it: temporarily set
`DATABASE_URL` in `.env.test` to a non-`_test` name and run `npm test` — the
suite fails immediately with a `[db-guard]` error and deletes nothing.

Test Redis is also isolated to logical DB `/1` (`redis://localhost:6379/1`), so
test login-lockout keys never collide with your dev Redis (`/0`).

## Troubleshooting

### "Session expired" / login fails / seed accounts missing

Symptom: you log in with `buyer@test.com` on the sim and it fails, often with a
misleading **"Session expired"** (a separate interceptor issue — see PR #132).
The real cause is usually one of: the seed account is missing, its password was
overwritten, or Redis has you locked out after 5 failed attempts.

**Diagnose first:**
```bash
cd backend && npm run db:doctor
```
It tells you exactly which of: accounts missing, wrong password, categories
missing, or an active Redis lockout.

**Recover:**
```bash
npm run db:reset-lockout   # clears auth:login_lockout:* and auth:login_attempts:* (keyed by email)
npm run db:seed            # re-creates/repairs the 4 seed accounts
npm run db:doctor          # confirm green
```

> Redis lockout keys are keyed by **email**: `auth:login_lockout:{email}` and
> `auth:login_attempts:{email}` (`MAX_LOGIN_ATTEMPTS=5`, 15-minute TTL — see
> `src/modules/auth/auth.service.ts`).

### Tests aborted with a `[db-guard]` error

The guard refused to run because `NODE_ENV` wasn't `test` or `DATABASE_URL`
didn't point at a `_test` database. Run `npm run db:bootstrap` to (re)generate a
correct `.env.test`. This is the guard working as intended — it protected your
dev data.

### I ran `npm test` on an old checkout and my seed accounts are gone

Re-seed: `npm run db:seed`. On this branch and later, `npm test` uses the
isolated test DB and can't do this anymore.
